The Chase transactions for May 14-28 are NOT lost. They're in the database, but the cleanup orphaned them.

ROOT CAUSE — confirmed by reading scopeChaseTransactions in artifacts/h2budget/src/lib/chaseEndingBalance.ts:53:

When chasePlaidAccountId === null (no Chase Plaid account linked, which is true now), the fallback filter is:
  t => !t.plaidAccountId && isChaseFallbackSource(t.source ?? null)

That requires `plaidAccountId` to be NULL. But the May 14-28 transactions were synced via Plaid BEFORE the cleanup ran — they have plaidAccountId set to the (now-deleted) Chase TOTAL CHECKING 5526 account uuid. The cleanup deleted the plaid_accounts row, but did NOT null out the plaidAccountId pointers on the transactions referencing it. So those rows are orphans: their plaidAccountId points at a row that no longer exists.

Result: fallback filter rejects them (plaidAccountId is non-null), id-based filter rejects them (no current account id matches the orphaned uuid). They're invisible.

FIX — TWO PARTS:

PART 1 — One-shot orphan repair.

In startupChaseDeadItemsCleanup.ts, after the plaid_items + plaid_accounts deletion loop completes, find all transactions in TARGET_HOUSEHOLD whose plaidAccountId points at an account that no longer exists in plaid_accounts. For those rows, UPDATE transactions SET plaidAccountId = NULL. Don't touch source, category, or any other field — just null out the orphan pointer.

Roughly:
  UPDATE transactions
    SET plaid_account_id = NULL
    WHERE household_id = TARGET_HOUSEHOLD
      AND plaid_account_id IS NOT NULL
      AND plaid_account_id NOT IN (SELECT id FROM plaid_accounts);

Log the count. Idempotent — runs once, finds zero on subsequent boots.

PART 2 — Make this part of every plaid_account delete going forward.

In whatever path deletes a plaid_accounts row (the cleanup endpoint, the per-item DELETE route, the startup script), null out the plaidAccountId on every transaction pointing at the soon-to-be-deleted account BEFORE deleting the account row. Wrap in transaction. Prevents this class of orphan from ever appearing again.

PART 3 — Also null out forecast_settings.bankSnapshotAccountId if it points at a deleted account (the "Plaid · TOTAL CHECKING ··5526 · snapshot" caption residue). Same orphan-pointer issue, different table.
