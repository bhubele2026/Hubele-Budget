There's a confirmed bug in artifacts/api-server/src/lib/dedupePlaidAccounts.ts that collapses two real Amex cards into one when they share a mask. My Platinum Card and Delta SkyMiles Gold both end in 1009 — the dedupe key (institutionName, mask) treats them as duplicates and deletes Delta Gold's plaid_accounts row, rewriting its transactions to point at Platinum.

CHANGE 1 — Tighten the dedupe key in artifacts/api-server/src/lib/dedupePlaidAccounts.ts.

Find line ~130-135:

  const key = `${(b.institutionName ?? "").toLowerCase()}|${b.acct.mask.toLowerCase()}`;

Replace with a key that ALSO includes the account name. Two cards with the same institution and same mask but DIFFERENT names are not duplicates — they are genuinely different cards (Amex masks are only 4 digits but card numbers actually differ in earlier positions, so distinct products can collide on mask).

  const acctName = (b.acct.name ?? b.acct.officialName ?? "").toLowerCase().trim();
  const key = `${(b.institutionName ?? "").toLowerCase()}|${b.acct.mask.toLowerCase()}|${acctName}`;

This means:
- Re-link case (same institution, same mask, same name, different account_id) → still gets deduped. Good.
- Same-mask-different-name case (Platinum 1009 vs Delta Gold 1009) → no longer collide. Fixed.
- Same-name-different-mask → never collided. Unchanged.

CHANGE 2 — Also tighten the trailing accountSnapshots backfill around line 425-444 in the same file.

It does a similar mask-only fallback when the name is missing:

  const entryMask = entry.mask?.toLowerCase() ?? null;
  const entryName = (entry.name ?? "").toLowerCase();
  ...
  (r) => (r.mask ?? "").toLowerCase() === entryMask

Make the matching prefer the (mask, name) pair when both are available, and only fall back to mask-only when no name is present on either side. The intent is the same as CHANGE 1 — don't conflate two real cards that happen to share a mask.
