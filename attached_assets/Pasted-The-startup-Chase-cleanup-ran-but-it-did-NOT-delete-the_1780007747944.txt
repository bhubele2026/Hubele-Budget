The startup Chase cleanup ran but it did NOT delete the live (OAuth-invalidated) b85526fb Chase item. Evidence: after Publish + hard refresh, the /chase page still shows "Plaid · TOTAL CHECKING ··5526 · Current balance $2,000.00 · snapshot", the reauth banner still fires, and a sync attempt JUST surfaced a toast "Sync had errors / Chase: The user's OAuth connection to this institution has been invalidated."

That toast means the api-server STILL has a Chase plaid_item with a parseable access_token, and is actively trying to sync it. So either:
(a) startupChaseDeadItemsCleanup never executed (boot path failure), OR
(b) it executed but the live item didn't match the deletion criteria

ROOT CAUSE — most likely (b):

Look at startupChaseDeadItemsCleanup.ts line 108:
  const dead = synthetic || tokenMalformed || tokenEnvMismatch || hasErrorCode;

For the OAuth-invalidated b85526fb item, this evaluates as:
  - synthetic: false (it's a real Plaid item, itemId doesn't start with "seed-")
  - tokenMalformed: false (the access_token is well-formed)
  - tokenEnvMismatch: false (token was minted for current PLAID_ENV)
  - hasErrorCode: depends on whether lastSyncErrorCode was populated when the sweep ran

If the sweep ran during a window where lastSyncErrorCode happened to be null (e.g. right after the initial balance refresh succeeded at 21:27, BEFORE the failing /transactions/sync at 21:46 stamped ITEM_LOGIN_REQUIRED on the row), all four flags would be false and the item would skip deletion.

There's also a secondary issue: the LEFT JOIN in listCheckingAccounts (forecast.ts line 127) means even if the plaid_item gets deleted, orphaned plaid_accounts rows still show up in the checking-account picker / snapshot caption. This is independent of the cleanup gap.

FIX — TWO PARTS:

PART 1: Widen the "dead" predicate in startupChaseDeadItemsCleanup.ts.

In addition to the existing four flags, treat an item as dead if its sync state is in any of these states regardless of lastSyncErrorCode:
  - consentExpirationAt is in the past
  - the item has zero attached plaid_accounts (orphaned shell)
  - the item is OAuth-invalidated per Plaid's API (do a single best-effort plaid().itemGet() and treat any error response or status.transactions.last_failed_update with login_required as "dead")

Specifically for the Chase b85526fb case: even with NO lastSyncErrorCode persisted on the row, an itemGet() probe will return error_code = ITEM_LOGIN_REQUIRED because Chase has revoked the consent. That probe is authoritative.

The probe is per-Chase-item-only, only fires at startup, and only when the existing four flags didn't already mark it dead — so cost is negligible (max 2 Plaid API calls at boot for my household).

PART 2: Fix orphan accounts.

Add to startupChaseDeadItemsCleanup.ts a second sweep AFTER the item-deletion loop:
  SELECT plaid_accounts WHERE household_id = TARGET AND item_id NOT IN (SELECT id FROM plaid_items)
  → these are orphans from any prior partial-delete. Delete them. Reset any debts pointing at them to manual sources (same downshift logic).

This is a permanent guard: even if a future cleanup partially fails, the next boot will sweep orphans.
